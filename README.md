# フロントエンド大量データ パフォーマンス検証アプリ

Next.js + TypeScript で、大量データをブラウザ上に生成し、フロントエンドだけでどこまで処理を改善できるかを比較する検証アプリです。

バックエンドやデータベースの最適化ではなく、ブラウザ内の実装だけで `filter`、`reduce`、`useMemo`、`Map`、`Index`、`Web Worker` がどの程度効くのかを実測します。

データは特定業界に依存しない抽象的な会社データです。100社程度、複数部門、60か月分のランダムな売上・顧客数・案件数・稼働率を生成します。

## 起動

```bash
npm install
npm run dev
```

ブラウザで `http://localhost:3000` を開きます。

## 検証できること

- 10,000 / 100,000 / 1,000,000 / 5,000,000 / 10,000,000 / 20,000,000 / 50,000,000 件のランダム会社データ生成
- 5,000,000 件までは `BusinessRecord[]`、10,000,000 件以上は TypedArray のカラム形式で保持
- 会社、部門、年月によるリアルタイムフィルター
- 配列 `filter` + `reduce` の基準ケース
- `useMemo` による同条件再描画時の再計算回避
- `find()` と `Map.get()` の lookup 差分
- 事前 Index による filter 走査件数削減
- 数千万件では集計キューブ型 Index により、全件スキャンとの差を比較
- Web Worker による UI ブロック軽減
- Recharts と Chart.js の同時表示による Chart 更新時間比較
- 棒グラフ、横棒グラフ、折れ線グラフ、レーダーチャートで異なる平均集計を可視化
- Performance API による filter / 平均計算 / chart更新 / 合計時間の履歴比較
- ランダム生成と Index 構築の進捗表示、経過時間表示、キャンセル

## 比較の見方

左パネルで改善機能を ON/OFF し、フィルターや再レンダーボタンを操作すると、画面下部の比較履歴に計測結果が残ります。

Worker は処理時間そのものを短縮する機能ではなく、メインスレッドの停止時間を減らすための比較対象です。履歴の `UI最大停止` を同期処理ケースと比較してください。

10,000,000 件以上はオブジェクト配列のままではブラウザメモリを消費しすぎるため、`companyId` や `revenue` などを TypedArray に分けて保持します。ローカル Next.js では COOP/COEP ヘッダーを付け、SharedArrayBuffer が使えるブラウザでは Worker とデータを共有します。

ランダムデータは会社規模、部門係数、月次の季節性、まれな売上スパイクを混ぜて生成します。単純な一様乱数よりも会社別売上や月別推移の差が出やすく、グラフ描画の比較が見やすくなります。

グラフは Recharts と Chart.js を切り替えるのではなく同時に描画します。部門別売上平均、会社別売上平均、月別稼働率平均、月別顧客数平均、部門別案件数平均、部門別稼働率平均を別々の表現で表示し、複数チャートを更新したときの描画コストも見られるようにしています。

## データ例

```ts
type BusinessRecord = {
  id: number;
  companyId: number;
  companyName: string;
  businessUnit: "営業" | "開発" | "マーケティング" | "カスタマーサクセス" | "管理" | "人事";
  month: string;
  revenue: number;
  customerCount: number;
  projectCount: number;
  utilization: number;
};
```

## このアプリで見たいこと

- 全件 `filter` は件数が増えるとどのくらい重くなるか
- `reduce` による平均・合計計算はどの程度ボトルネックになるか
- `useMemo` は同条件の再計算回避にどれだけ効くか
- `Map.get()` は `find()` よりどの程度速いか
- 事前 Index 化で走査件数をどれだけ減らせるか
- Web Worker で UI の停止感をどれだけ減らせるか
